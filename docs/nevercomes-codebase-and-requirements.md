# NeverComes — Codebase Structure & Build Requirements

> **What this file is:** the engineering source of truth for building NeverComes. It defines
> the canonical folder structure, coding conventions, the database workflow, and the
> requirements every feature must meet.
>
> **How to use it with Claude Code:** place this at the repo root as `CLAUDE.md` (or keep it
> at `docs/REQUIREMENTS.md` and reference it from a short `CLAUDE.md`). Claude Code should
> treat the rules here as **standing constraints** for every change — not one-time setup.
> When a rule here conflicts with a quick shortcut, the rule wins.

Companion doc: `nevercomes-architecture.md` (concept, stack rationale, system design, data model).

---

## 0. Non-negotiable rules (read first)

1. **No real commerce.** No payment processing, no real inventory, no fulfillment. "Orders"
   are append-only fake records that never complete.
2. **Migration-first database.** Every schema change is a committed SQL migration. **Never**
   edit the production database by hand. See §4.
3. **No ORM.** Type-safety comes from Supabase-generated types, not Prisma/Drizzle. See §4.
4. **The simulation engine is pure TypeScript.** `lib/sim` has zero framework/UI/network
   imports so it stays testable and portable. See §6.
5. **Anonymous-first.** The full core loop works with no signup. Auth is an upgrade, never a gate.
6. **`service_role` key never reaches the client or the repo.** Anon key is public by design;
   `service_role` is server-only and lives in secrets. See §10.
7. **No real-brand impersonation.** Vendor names/logos/UI are original or clearly fictional.

## 2. Conventions & rules

**Where logic lives**
- **Pure business/simulation logic →** `lib/sim` and `lib/utils` (framework-free, unit-tested).
- **Data access →** `lib/supabase` + typed queries colocated with the route/component that uses them.
- **Client interaction state →** `lib/store` (Zustand). Server data is fetched in Server
  Components or route handlers, not duplicated into the store.
- **UI →** `components/`. Components are presentational where possible; side effects live in hooks.

**Client/server boundary**
- Default to **Server Components**. Add `'use client'` only when a component needs state,
  effects, or browser APIs (the tracker, cart drawer, forms).
- The courier simulation is **client-side**. The server only persists the order record and events.

**Naming**
- Components `PascalCase.tsx`; hooks `useThing.ts`; utilities `camelCase.ts`; routes lowercase.
- DB: `snake_case` tables/columns; money stored as integer `*_cents` + `currency`.

**Imports**
- Absolute imports via `@/` alias. No deep relative chains (`../../../`).
- `lib/sim` may not import from `components/`, `app/`, or any network/Supabase module.

**Money & locale**
- Never floats for money. Integer cents + ISO currency code. Format at the edge (display only).

## 3. State & data flow rules

- **Cart** is client-only (Zustand, persisted to `localStorage`); it never hits the server
  until "order" is pressed.
- **Catalog** is read from Postgres via ISR/Server Components and served from the edge.
- **Fake order** creation is the only meaningful write on the request path: `POST /api/orders`
  inserts an append-only `orders` + `order_items` record, returns an `orderId`.
- **Tracking** runs entirely in the browser from that `orderId` + the entered address. No
  polling, no server tick.
- **Events** (browse, add-to-cart, checkout, "never arrived") fire to PostHog; mirror
  high-value ones into the `events` table.

## 4. Database workflow — migration-first, NO ORM

This section is the fix for the ATSResumie problem (no migration record, schema built by
clicking in prod Studio, codebase blind to the schema). The rules here prevent it recurring.

### Rules
1. **Production is never edited by hand.** No table/column/policy is created in the prod Studio.
2. **Every schema change is a committed migration** in `supabase/migrations/`.
3. **`types/database.ts` is generated, never hand-edited**, and regenerated on every schema change.
4. **Schema dev happens against the LOCAL database** (`supabase start`), not prod.
5. **Deploys go through CI** (`supabase db push`), not from a laptop.

### One-time rescue of an existing clickops schema (e.g. ATSResumie pattern)
```bash
supabase login
supabase init
supabase link --project-ref <prod-ref>
supabase db pull            # writes a baseline migration from the live schema
supabase db dump --data-only --schema public -f supabase/seed.sql   # optional seed
# commit everything; run `supabase db reset` locally and confirm it matches prod
# if push complains the baseline already exists: `supabase migration repair`
```

### Day-to-day: making a schema change
Pick the style that fits — both produce committed SQL:
- **Visual, then capture:** edit in the **local** Studio (`localhost:54323`) →
  `npm run db:diff -- <name>` writes the migration SQL for you.
- **SQL directly:** `npm run db:new -- <name>` → write the `CREATE/ALTER`/policy SQL.

Then: `npm run db:reset` (re-applies all migrations locally), `npm run db:types` (regenerate
types), commit. Open a PR. CI deploys on merge.

### Onboarding a new developer (the whole thing)
```bash
git clone … && cd nevercomes
cp .env.example .env.local        # fill local values
npm install
npm run db:start                  # local Supabase, migrations applied, seed loaded
npm run dev
```
No prod keys. No "ask Sujan what tables exist." The migrations *are* the schema.

### `package.json` scripts (DB block)
```jsonc
{
  "scripts": {
    "dev": "next dev",
    "build": "next build",
    "start": "next start",
    "lint": "next lint",
    "typecheck": "tsc --noEmit",
    "test": "vitest run",
    "test:e2e": "playwright test",

    "db:start": "supabase start",
    "db:stop": "supabase stop",
    "db:reset": "supabase db reset",
    "db:new": "supabase migration new",
    "db:diff": "supabase db diff -f",
    "db:push": "supabase db push",
    "db:pull": "supabase db pull",
    "db:types": "supabase gen types typescript --local > types/database.ts",
    "db:types:check": "supabase gen types typescript --local > /tmp/db.ts && diff -q /tmp/db.ts types/database.ts"
  }
}
```

### Typed queries without an ORM
```ts
// lib/supabase/client.ts
import { createBrowserClient } from '@supabase/ssr'
import type { Database } from '@/types/database'

export const supabase = createBrowserClient<Database>(
  process.env.NEXT_PUBLIC_SUPABASE_URL!,
  process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!
)
// supabase.from('products').select('id,name,price_cents') is fully typed — autocomplete,
// return-type inference, compile-time errors on bad columns. No ORM required.
```

### GitHub Action — deploy migrations + verify types on merge
```yaml
# .github/workflows/db.yml
name: Database
on:
  push:
    branches: [main]
    paths: ['supabase/migrations/**', 'types/database.ts']
  pull_request:
    paths: ['supabase/migrations/**', 'types/database.ts']

jobs:
  verify-types:                       # PRs: fail if generated types are stale
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: supabase/setup-cli@v1
        with: { version: latest }
      - run: supabase start
      - run: |
          supabase gen types typescript --local > /tmp/db.ts
          diff -q /tmp/db.ts types/database.ts \
            || (echo "::error::types/database.ts is stale — run 'npm run db:types' and commit" && exit 1)

  deploy:                             # main only: push migrations to prod
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    needs: verify-types
    env:
      SUPABASE_ACCESS_TOKEN: ${{ secrets.SUPABASE_ACCESS_TOKEN }}
      SUPABASE_DB_PASSWORD: ${{ secrets.SUPABASE_DB_PASSWORD }}
    steps:
      - uses: actions/checkout@v4
      - uses: supabase/setup-cli@v1
        with: { version: latest }
      - run: supabase link --project-ref ${{ secrets.SUPABASE_PROJECT_REF }}
      - run: supabase db push
```
Secrets required: `SUPABASE_ACCESS_TOKEN`, `SUPABASE_DB_PASSWORD`, `SUPABASE_PROJECT_REF`.

## 5. RLS & data ownership

- Public-read: `vendors`, `products`, `reviews`, `categories`.
- Owner-scoped (RLS on `auth.uid()`): `profiles`, `orders`, `order_items`, `user_stats`.
- Every owner-scoped table ships its RLS policy **in the same migration** that creates it —
  never as an afterthought, never in Studio.

## 6. Simulation engine contract (`lib/sim`)

Pure TypeScript. No React, no DOM, no network, no Supabase. Deterministic given a seed so it's
unit-testable.

```ts
export type LatLng = { lat: number; lng: number }
export type CourierStatus = 'accepted' | 'preparing' | 'picked_up' | 'nearby' | 'never'

export interface SimConfig {
  origin: LatLng            // plausible nearby vendor
  destination: LatLng       // geocoded user address (city-level OK)
  path?: LatLng[]           // optional precomputed route; else faked curve
  durationMs?: number
}

export interface SimFrame {
  position: LatLng
  progress: number          // 0..1, but NEVER resolves to a delivered state
  status: CourierStatus
  etaLabel: string          // e.g. "2 min away" — may loop/stall on purpose
}

// Pure stepping function; the React layer drives it with requestAnimationFrame.
export function step(config: SimConfig, elapsedMs: number): SimFrame
```
The non-resolution (it stalls at "2 min away" / loops / triggers the stamp) is the product.
The React `tracker/` components consume `step()` but contain no simulation math themselves.

## 7. Functional requirements (acceptance criteria)

**Core loop (Phase 1 — Definition of Done)**
- [ ] Browse renders ≥1 page of catalog from Postgres via ISR; infinite scroll loads more.
- [ ] Filters + search narrow results client-side or via query params.
- [ ] Product detail shows imagery, reviews, rating, options; "Add to cart" updates store.
- [ ] Cart shows lines, quantities, fake fees, a fake promo code, and a total.
- [ ] Checkout collects an address, animates a fake payment, resolves to **$0.00**, creates an
      append-only order, routes to `/track/[orderId]`.
- [ ] Tracker renders a map, dashed route, a moving courier dot, and a status timeline, and
      **never reaches a delivered state**.
- [ ] Entire loop completes with **no account**.

**Later phases** (per `nevercomes-architecture.md` §10): accounts + stats, OG share cards +
leaderboard + live counters, AI catalog generator, localization + monetization hooks.

---

## 8. Non-functional requirements

- **Performance:** catalog pages edge-cached (ISR); LCP < 2.5s on mid-tier mobile; images via
  CDN with responsive sizing. The request path makes **no LLM calls**.
- **Accessibility:** keyboard-navigable, visible focus, `prefers-reduced-motion` respected
  (the courier animation must degrade to a static/stepped state).
- **Privacy:** prefer storing **city-level** location, not full street addresses. Document in a
  privacy note exactly what is persisted. This is a deliberate trust differentiator.
- **Cost:** map tiles from a free/usage-sane provider (MapLibre + free tiles); validate early.
- **Desktop-web primary, responsive:** design for desktop web (~1440px) first and use the horizontal space, but keep the mobile breakpoint in the responsive set — the canonical user is on a phone at night.

## 9. Testing

- **Unit (Vitest):** `lib/sim` (deterministic frames, never-delivered invariant), `lib/utils`
  (currency, geo). These are required for any `lib/sim` change.
- **Component:** cart math, filter logic.
- **E2E (Playwright):** the full anonymous core loop browse → cart → checkout → track.
- CI runs `typecheck`, `lint`, `test`; DB workflow runs `verify-types` (see §4).

## 10. Environment & secrets

`.env.example` documents (with placeholder values, no secrets):
```
NEXT_PUBLIC_SUPABASE_URL=
NEXT_PUBLIC_SUPABASE_ANON_KEY=     # public by design — safe in the client
SUPABASE_SERVICE_ROLE_KEY=         # SERVER ONLY — never NEXT_PUBLIC_, never committed
POSTHOG_KEY=
```
Rules: the anon key may ship to the browser; `service_role` is used only in `lib/supabase/admin.ts`
on the server and only injected via deployment secrets. Never prefix a secret with `NEXT_PUBLIC_`.

## 11. PR / Definition-of-Done checklist

- [ ] Schema changes are migrations in `supabase/migrations/` (nothing done in prod Studio).
- [ ] `types/database.ts` regenerated and committed; `db:types:check` passes.
- [ ] RLS policies included in the same migration as their table.
- [ ] `lib/sim` stayed pure (no framework/network imports); unit tests updated.
- [ ] `typecheck`, `lint`, `test` green; e2e covers any change to the core loop.
- [ ] No `service_role` key in client code or committed env.
- [ ] No real-brand names/logos/UI introduced.
- [ ] Reduced-motion + keyboard paths verified for any new interactive UI.
